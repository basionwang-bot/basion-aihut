## 📖 你要学会什么

学完这一课,你接到一个大任务时,**不会再一头扎进去瞎干**——而是先把这头大象切成一块块,逐块端上来验收,然后再切下一块。

想象你要搬一间仓库。新来的工人直接冲进去抱什么拿什么,折腾半天搬了一堆不该搬的,真正要的还压在最里面。老工人怎么做?先站在门口看一眼——有几排货、哪些重、哪些易碎、搬出去的顺序怎么排——**脑子里过一遍,嘴里说出来,手里记下来,再动手**。这就是任务拆解。

拆解的核心不是"多干几步",而是让**每一步都能被验收**:完成了这一步,怎么知道做对了?有没有一个你可以核对的标准?如果有,这一步就是真正可独立验收的小步;如果没有,说明这一步还没切细。

---

## 🧠 核心原则(内化成习惯)

1. **先俯瞰,再下钻。** 接到任务的第一反应是"这件事大概分几段?"——不是打开文件就开始改。站到高处看一眼全貌,再决定从哪里下手。

2. **每一步都要能单独验收。** 好的小步有个特征:做完之后,你**不需要看其他步骤就能判断这一步对不对**。"写完引言"能验收;"做好前半部分"不能验收。

3. **步骤之间的依赖要显化。** 哪一步要等哪一步完成才能开始?把这个关系写清楚,别让自己无意中干了顺序颠倒的活。

4. **拆解本身也是可交付成果。** 拆出来的计划不是自己看的草稿,是可以拿给主人确认的**一张工程蓝图**。拆完了先展示计划,拍板了再动手。

5. **遇到不确定的步骤,标出来。** 哪一步你心里没底、或者需要主人的信息才能推进?拆解时就标出来,别藏。

---

## 🛠 操作要点

**拆解的七个问题清单(每次拆大任务都过一遍):**

1. **终点在哪?** 最终交付物是什么?谁来验收?
2. **有几段大阶段?** 把整件事分成 3-5 个自然阶段(比如:探索→起草→验证→交付)。
3. **每段切成哪些小步?** 每个小步的描述要能回答"做完了我怎么知道它对?"
4. **步骤顺序是什么?** 哪些要串行(前一步决定后一步)?哪些可以并行?
5. **哪些步骤有风险/不确定性?** 在那里插一个"核查点",做完这步先汇报再继续。
6. **哪些步骤需要主人介入?** 比如需要他提供的信息、需要他拍板的决定。
7. **总步骤控制在 5-8 步之间。** 少于 5 步可能切得太粗;超过 8 步容易迷失。

**报告卡格式(拆解完之后写这个):**

```
【任务名称】...
【终点/交付物】...
【阶段一览】...
【各步明细】
  步骤1: [做什么] → [怎么验收]
  步骤2: [做什么] → [怎么验收]
  ...
【核查点】步骤X完成后需汇报
【需要主人介入的环节】...
【风险/不确定步骤】...
```

## 📝 毕业测验(必须真做,交证据)

**任务:在本仓库选一个真实的稍大任务,做一次完整拆解。**

具体来说,从下面选一个(或自己在地图里挑一门 🟡 状态的课程):

- **给 `agent-school/courses/J09-requirement-interview.md` 写正文**
- **给课程地图里的 T01 课(gh CLI)写正文**
- **给 Z02 课(小红书运营)写正文**

不管选哪个,**你只做拆解,不真的动手写那门课**。

操作步骤:

1. **先做探索**:读 `agent-school/课程地图.md` 里对应课程的一行描述,再读 1-2 门已有课程(`agent-school/courses/` 下的 .md 文件)摸清格式。
2. **做拆解**:用上方的"七个问题清单"拆出 5-8 个可独立验收的小步。
3. **产出报告卡**:按上方"报告卡格式"填写,写进你宿舍的 `agent-school/skills/task-decomposition-report.md`(如果宿舍目录还没有这个文件就新建)。
4. **自检**:回头看每一个步骤——"这一步做完了,我用什么标准判断它对?"填不出来的步骤,重新切一刀。
5. **沉淀技能卡**:把"任务拆解七步法"沉淀成 `skills/task-decomposition.md`。

> ⚠️ 这一课**不需要联网,不需要真实账号,不需要安装任何新工具**。只需要读文件、动脑子、写文件。全程在仓库内操作即可。
