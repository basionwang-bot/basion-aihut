## 📖 你要学会什么

你有没有羡慕过 Zapier 那种"当 A 发生时,自动做 B"的魔法?每收到一封邮件就自动存表格,每发一条微博就自动同步到其他平台……

**Automatisch 就是 Zapier 的自托管开源版**——把那套魔法装进你自己的服务器。你的数据不流出去,不被别人的云存储,不用担心隐私泄漏。对于处理敏感数据、想自己管数据的场景,这是极好的选择。

诚实说明:**Automatisch 是成长期项目**。目前约有数十个集成连接器(远少于 Zapier 的 6000+),活跃度中等(约 3 个月一个版本,v0.15.0 发布于 2025 年 8 月)——功能在持续增加,但不像 n8n 那样生态成熟。选它的核心理由是:**最简单、最像 Zapier、最适合第一次上手自动化的人**。

学完这一课,你能:
- 用 Docker 在本地 5 分钟内把 Automatisch 跑起来
- 理解"触发器(Trigger)→动作(Action)"这对核心概念
- 搭出你的第一个"A 发生就做 B"的自动化工作流

## 🧠 核心原则

1. **触发器 + 动作 = 自动化的全部逻辑。** 想象你在家里装了一个感应灯:有人经过(触发器)→ 灯亮(动作)。Automatisch 的每一条自动化就是这样:一个"什么时候/什么情况下"的触发器,加上一个或多个"然后去做什么"的动作。就这两件事。

2. **数据留在你家。** Automatisch 的核心价值主张:所有数据存在你自己的服务器上,满足 GDPR 合规要求。如果主人对数据隐私敏感(比如处理客户信息、内部文件),Automatisch 比云端服务更安全。

3. **许可证是 AGPL-3.0,使用要了解含义。** AGPL-3.0 是开源许可证,但有传染性——如果你基于它修改并对外提供服务,你的修改也必须开源。**个人自用、内部使用**完全没问题;如果要商业对外服务,先问主人。企业版有商业许可证可选。

4. **连接器数量有限,用前核查。** Automatisch 目前连接器比 n8n、Zapier 少得多。接入某个平台之前,先到官网查一下是否支持,不要假设"应该有"。

5. **先征得主人确认,再安装、再接账号。** Docker Compose 会在本地创建数据库和服务进程,需要主人确认。接外部 OAuth 账号(GitHub、Slack、Twitter 等)时,必须由主人完成 OAuth 授权流程。

## 🛠 操作要点

### 用 Docker Compose 启动 Automatisch

Automatisch 官方推荐用 Docker Compose 部署。先确认 Docker 和 Docker Compose 已安装:

```bash
docker --version
docker compose version
```

> ⚠️ **安装 Docker 前先征得主人确认。**

拉取代码并启动:

```bash
git clone https://github.com/automatisch/automatisch.git
cd automatisch
docker compose up
```

等待镜像下载和容器启动(首次可能需要几分钟)。看到类似 `Server is running on port 3000` 的日志,表示启动成功。

浏览器打开:

```
http://localhost:3000
```

**默认登录账号:**
- 邮箱: `user@automatisch.io`
- 密码: `sample`

> ⚠️ **登录后立即修改密码!** 进入 Settings → Profile 改掉默认密码。不改就是把大门开着。

> 🇨🇳 **国内友好提示:** Automatisch 本身完全本地运行,无需科学上网。Docker 镜像如果拉取慢,可以配置 Docker 镜像加速器(如 DaoCloud、阿里云镜像加速)。

### 理解界面结构

登录后,你看到的主界面大致是:

```
左侧导航:
├── Flows (工作流列表)
├── Connections (已连接的外部账号)
├── Admin (管理设置)
└── ...

中间内容区:
- 工作流编辑器:触发器 → 动作 → 动作...
```

- **Connections**:先在这里把外部服务的账号授权连接好(类似"插好插头")
- **Flows**:再在这里创建工作流,引用已连接的账号

### 连接外部服务账号

在创建工作流之前,先把需要的账号连起来:

1. 进入左侧 **Connections** → **Add connection**
2. 选择服务(如 GitHub、Slack、Twitter 等)
3. 点 **Connect**,会跳转到该服务的 OAuth 授权页面
4. **由主人完成授权**——这一步你不能代劳,因为需要主人的账号密码

> ⚠️ **OAuth 授权必须由主人操作。** 你可以帮主人找到入口、说清楚每一步在做什么,但不能替主人输入账号密码或点击授权。

### 创建第一个自动化工作流

以"GitHub 有新 Issue → 推送 Slack 通知"为例(假设主人已连接这两个账号):

**步骤一:新建 Flow**

1. 左侧 **Flows** → **Create Flow**
2. 给 Flow 起个名字,如"GitHub Issue 提醒"

**步骤二:设置触发器**

1. 点击画布上的"+"选择触发器
2. 选择 **GitHub** → **New Issue**
3. 选择你已连接的 GitHub 账号
4. 选择要监控的仓库
5. 保存

**步骤三:设置动作**

1. 在触发器下方点"+"添加动作
2. 选择 **Slack** → **Send Message**
3. 选择已连接的 Slack 账号和目标频道
4. 在消息内容里,可以用触发器传来的数据,如 `{{issue.title}}` 等变量
5. 保存

**步骤四:开启 Flow**

编辑完成后,找到 Flow 的开关,把状态切换到 **Active**。Flow 就开始监听了。

> ⚠️ **开启 Flow 前先确认主人知情**:Active 状态下,Flow 会真实调用外部服务。如果是测试阶段,先确保触发条件不会意外产生大量噪音。
