> **本模块教学目标：** 建立正确的心智模型。这是后面所有技巧的前提——心态没换对，技巧学了也用不出来。

---

## 为什么"换脑子"要放在第一课？

因为大多数人用不好 Claude Code，**不是技术问题，是心态问题。**

他们带着"用聊天机器人"的习惯来：
- 问一句、等一句、复制走人
- 一次问题没答好就觉得"AI 不行"
- 把它当搜索引擎，或者当代码生成器

结果当然差。这就像你雇了个能干的助理，却只把他当自动售货机用——投币、出货、走人，那他一身本事全浪费了。

**先把脑子里的比喻换对，是这门课最重要的一步。**

---

## 错误比喻 vs 正确比喻

| ❌ 你以为它是 | ✅ 它其实是 |
|--------------|------------|
| 聊天机器人（你问它答） | 一个**能动手干活**的初级工程师 |
| 代码生成器（吐代码片段） | 一个会**自己探索、自己计划、自己实现**的人 |
| 搜索引擎（查信息） | 一个有**记忆、能迭代、会用工具**的同事 |

记住这个正确比喻：

> **🛠 Claude Code = 一个带着全套工具、有记忆、能迭代的初级工程师。**

"初级"两个字很关键——它聪明、勤快、工具齐全，但**需要你给清楚的目标和验收标准**。你不会对一个新来的实习生说"把项目搞好"就走人，对吧？你会说清楚"做什么、做成什么样、怎么算完成"。对 Claude 也一样。

---

## 工作方式的根本转变

老习惯（你是主力，AI 是辅助）：

> **你写代码 → 它帮你审一下 / 补两行**

新方式（它是主力，你是主管）：

> **你描述目标 → 它去探索 → 它出计划 → 它来实现 → 你验收**

注意角色变了：**你从"操作工"升级成了"管理者"。** 你的核心工作不再是"自己动手"，而是：

- 把目标讲清楚（模块二）
- 给它能自检的标准（模块四）
- 管好它的注意力（模块五）
- 在它跑偏时及时纠正

这就是为什么这门课的主线是"把自己活成一个团队"——**你当队长，Claude 们当队员。**

---

## 它具体"能"干什么？（新手常低估的部分）

很多新手不知道，Claude Code 是可以**自主推进任务**的。它能：

- **读你的文件**：你不用一个个贴给它，它会自己去翻整个项目
- **跑命令**：装依赖、运行程序、跑测试、提交 git——它都能自己执行
- **改代码**：直接修改你的文件，不是只给你"参考代码"
- **连续迭代**：跑测试 → 看到失败 → 自己分析 → 改 → 再跑，循环到通过

而你呢？**可以盯着看，也可以随时打断纠偏，或者干脆走开喝杯咖啡。**

---

## ⚠️ 同时引入一个贯穿全课的概念：上下文 = 工作记忆

既然它能读文件、跑命令，那就有个代价你必须知道：

> **它读过的每个文件、跑过的每条命令的输出、你们对话的每句话，全都占用它的"工作记忆"（上下文窗口）。**

一次稍微复杂点的调试或代码探索，**就可能消耗几万个 token**（你可以粗略理解为"几万字的记忆空间被占掉了"）。

为什么现在就要提这个？因为它会改变你的行为：

- 你会开始**舍不得**让它漫无目的地读几百个文件（那是在烧记忆）
- 你会开始**主动**给它划定范围
- 你会理解为什么后面模块五要专门讲"怎么清理记忆"

先有这个意识就够了，具体操作模块五细讲。

---

## 🎬 现场演示（课堂这样讲最有冲击力）

打开一个**你完全没见过**的开源项目，只问一句：

```text
这个项目是做什么的？
```

然后**什么都别做，看着它**：

- 它会自己去翻 README、看目录结构、读几个关键文件
- 然后给你一份**结构化的解读**：这是个什么项目、用了什么技术、核心模块在哪

整个过程你没贴任何文件、没给任何提示。学员第一次看到这个，会有一个直观的"卧槽，它真的会自己干活"的体感——**这个体感，比讲十句道理都管用。**

---

## ✍️ 学员练习（今天就动手）

在你自己的项目里，问 5 个"**你会问一位资深同事**"的问题。比如：

1. "这个项目的日志是怎么打的？我想加一条日志该照着哪里写？"
2. "如果我要新增一个页面，要动哪些文件？"
3. "这段代码为什么要这么写？背后有什么考虑？"
4. "这个项目跑测试的命令是什么？"
5. "我想改 X 功能，相关逻辑在哪几个文件里？"

**重点不是答案对不对，而是体会那种"在跟一个懂行的同事对话"的感觉。** 这个感觉建立起来了，模块一的目标就达到了。
