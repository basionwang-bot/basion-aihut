> **本模块教学目标：** 搭建个人 / 团队的 Claude Code 工作环境。前面五个模块教"怎么用得好"，这一模块教"怎么让它默认就好用"。

---

## 为什么要配置环境？

前面学的技巧，每次都要你手动施展。**环境配置的意义在于：把好习惯固化下来，让它默认就照着做。** 配一次，往后每个会话自动生效——这是"省力"的杠杆。

这一模块工具有点多，新手不用全上。**优先掌握前两个（CLAUDE.md 和权限），其余按需了解。**

---

## 6.1 CLAUDE.md —— 项目的"记忆文件"

这是最重要的一个配置，**新手必学。**

它是什么？一个放在项目里的 Markdown 文件，**每次会话开始 Claude 都会自动读它。** 你把"这个项目的规矩"写进去，它就不用每次你重复交代。

怎么创建：输入 **`/init`**，它会自动分析你的项目、生成一份初稿。然后你**持续打磨**它。

### ⚠️ 最重要的一条原则：保持简短

> **文件越臃肿，它越会忽略你的规则。** 这是最常见的配置失误。

判断标准就一句话：**"删掉这行，它会犯错吗？不会，就删。"**

下面这张表是黄金标准——**对照着写，别越界：**

| ✅ 该写进 CLAUDE.md | ❌ 别写进去 |
|---------------------|------------|
| 它猜不到的 Bash 命令（比如特殊的构建命令） | 它读代码就能看出来的东西 |
| 和默认不同的代码风格 | 语言通用规范（它本来就懂） |
| 测试命令和偏好 | 详细 API 文档（给个链接就行） |
| 分支命名、PR 规范 | 经常变动的信息 |
| 项目特有的架构决策 | "写干净的代码"这种正确的废话 |
| 环境怪癖（某个必需的环境变量） | 逐个文件的代码库说明 |

### 几个实用技巧

- 可以用 `@路径` **导入其他文件**，把内容拆开管理。
- 关键规则前面加 **"IMPORTANT"** 或 **"YOU MUST"** 来强调，它会更当回事。
- **放在哪里？**
  - `~/.claude/CLAUDE.md` —— 全局，你所有项目都生效
  - 项目根目录的 `CLAUDE.md` —— 进 git，**团队共享**
  - `CLAUDE.local.md` —— 个人专属，记得加进 `.gitignore`，不污染团队
  - 大型项目（monorepo）的父子目录会**按需自动加载**对应的 CLAUDE.md

## 6.2 权限三件套 —— 减少烦人的"点点点"

新手很快会遇到一个烦恼：它每干一步都弹窗问你"能不能执行这个命令？"，点到手软。三个办法治它：

1. **Auto mode（自动模式）** —— 有一个独立的"分类器模型"帮你审查命令，**只拦截看起来危险的**（越权操作、动了未知的基础设施、被恶意内容驱动的可疑行为）。安全的它自动放行。
2. **`/permissions` 白名单** —— 把你**确定安全**的命令加进白名单，以后不再问。比如 `npm run lint`、`git commit`。
3. **`/sandbox` 沙箱** —— 操作系统级别地隔离文件系统和网络，**给它划一个安全的院子**，院子里随便折腾，越不出边界。

新手建议：**先把你天天用、又绝对安全的几个命令用 `/permissions` 加白**，体验立刻清爽。

## 6.3 Skills（技能）—— 按需加载的领域知识 + 可复用流程

**和 CLAUDE.md 的分工，是理解 Skills 的关键：**

- **CLAUDE.md** —— 每次都加载。所以只放**普适规则**（放多了占记忆）。
- **Skills** —— 只在**相关场景**才加载。所以放那些"特定时候才用到"的领域知识和流程。

怎么建：在 `.claude/skills/技能名/SKILL.md` 里写。相关时它自动应用，或者你用 `/技能名` 手动调用。

知识型技能的例子（一份 API 规范）：

```markdown
---
name: api-conventions
description: 我们服务的 REST API 设计规范
---
# API 规范
- URL 路径用 kebab-case
- JSON 属性用 camelCase
- 列表接口必须带分页
- 版本号放 URL 路径（/v1/、/v2/）
```

这样，只有在它真的在写 API 时，这份规范才被加载进来，平时不占记忆。

## 6.4 Hooks（钩子）—— 确定性自动化

理解 Hook，记住这个**关键区别**：

> **CLAUDE.md 里的指令是"建议性"的——它大概率会照做，但不保证。Hook 是"确定性"的——每次必然执行，零例外。**

什么意思？比如你想"每次改完文件都跑一下 eslint"。写在 CLAUDE.md 里，它**可能**忘；做成 Hook，它**必然**执行。

好消息：**你可以直接让 Claude 帮你写 Hook：**

```text
写一个 hook，每次改完文件自动跑 eslint
```

用 **`/hooks`** 可以查看你已经配置了哪些钩子。（具体的"拿来即用"配方，模块七会给你七个。）

## 6.5 MCP —— 连接外部工具

MCP 让 Claude **接上你的其他系统**：Notion、Figma、数据库、监控系统……

用 `claude mcp add` 添加。接上之后能干嘛？

- 直接从一张 issue 单实现功能
- 查数据库
- 分析监控数据
- 照着 Figma 设计稿写界面

新手了解即可，等你有具体需求（比如"想让它直接读我 Notion 里的需求"）再来配。

---

## 6.6 自定义子代理 + 插件

- **自定义子代理**：在 `.claude/agents/` 里定义专职助手（比如一个"安全审查员"）。每个有**独立的上下文和独立的工具权限**，还能指定用哪个模型。
- **插件**：用 `/plugin` 浏览插件市场。**插件 = 技能 + 钩子 + 子代理 + MCP 打包安装**，一键装好一整套。

---

## 6.7 一个省力小习惯：优先用 CLI 工具

装上 `gh`（GitHub CLI）、`aws`、`gcloud` 这类命令行工具——**CLI 是最省上下文的外部交互方式**（一条命令搞定，不用读一堆网页）。

而且它还能**现学**没用过的 CLI：

```text
先跑 `foo --help` 学会这个工具，然后用它完成 A、B、C
```

## ✍️ 学员练习

今天就动手做两件事（新手版）：

1. 在你的项目里跑 `/init` 生成 CLAUDE.md，然后**对照那张黄金标准表，无情删减**，删到每一行都"删了它就会犯错"为止。
2. 把你最常用、最安全的 2-3 个命令用 `/permissions` 加进白名单，告别"点点点"。

进阶的（Skills / Hooks / MCP），等你有了具体痛点再回来配。

## 本模块一句话总结

> **配置的本质是把好习惯固化。** CLAUDE.md 管"每次都要的规则"（且越短越好），Skills 管"按需的知识"，Hooks 管"必须确定执行的动作"，MCP 管"连外部世界"。先把前两个配好，你已经超过大多数人了。
