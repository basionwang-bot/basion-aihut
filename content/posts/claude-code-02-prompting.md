> **本模块教学目标：** 学会写"第一次就能跑通"的指令，把返工次数降到最低。

---

## 一个朴素的真相

你给的指令越模糊，它就越要靠猜。猜错了，你就得返工；返工又污染了它的工作记忆（那张桌子又堆满了失败方案）。于是越改越乱，恶性循环。

> **提问的精度，直接决定返工的次数。** 花 30 秒把话说清楚，省下半小时的来回扯皮。

这一篇就教你怎么"把话说清楚"。

---

## 四个改写模板（看前后对比，秒懂）

下面四种情况覆盖了 90% 的日常提问。每一种都给你一个"从模糊到精确"的改写示范。

### 策略 1：限定范围

| | |
|---|---|
| ❌ 模糊 | "给 foo.py 加测试" |
| ✅ 精确 | "给 foo.py 写测试，**覆盖用户未登录的边界情况**，**不要用 mock**" |

**为什么更好？** "加测试"它可以有一百种理解。你一限定"覆盖什么场景、用不用某技术"，它就没得猜了。

### 策略 2：指向信息源

| | |
|---|---|
| ❌ 模糊 | "这个 API 怎么这么奇怪？" |
| ✅ 精确 | "**翻一下 ExecutionFactory 的 git 历史**，总结这个 API 是怎么一步步演变成现在这样的" |

**为什么更好？** 你告诉它"去哪儿找答案"，它就不用瞎猜，直接去翻历史记录给你有理有据的解读。

### 策略 3：引用现有模式

| | |
|---|---|
| ❌ 模糊 | "加一个日历组件" |
| ✅ 精确 | "**先看首页现有组件怎么实现的，HotDogWidget.php 是个好例子**，照这个模式做一个日历组件，**不要引入新依赖**" |

**为什么更好？** 这是新手最容易忽略、却最有用的一招。给它一个"照着抄"的范本，产出立刻和你项目的风格一致，而不是它自己发明一套。

### 策略 4：描述症状（而不是让它猜病因）

| | |
|---|---|
| ❌ 模糊 | "修一下登录 bug" |
| ✅ 精确 | "用户反馈**会话超时后登录失败**。查 `src/auth/` 里的 token 刷新逻辑。**先写一个能复现问题的失败测试，再修复它**" |

**为什么更好？** 你描述清楚"什么情况下、出了什么问题、可能在哪、怎么验证修好了"，它就能直奔根因，而不是头痛医头。

> 📌 看出规律了吗？四个模板的共同点都是：**把"你脑子里默认知道、但没说出口"的信息，明确写出来。** 你觉得理所当然的上下文，它并不知道。

## 五种"喂料"方式（怎么把信息给它）

光会提问还不够，你得知道怎么把资料递到它手上。一共五种：

1. **`@文件名`** —— 直接引用文件，它会先读再答。
   ```text
   照着 @src/components/Button.tsx 的写法，做一个 Card 组件
   ```
2. **截图 / 图片** —— 直接粘贴或拖进去。报错截图、设计稿、表格照片都行。
3. **文档 URL** —— 给它网址让它读。常用域名可以用 `/permissions` 加进白名单，省得每次问你要不要访问。
4. **管道灌数据** —— 把命令输出直接喂给它：
   ```bash
   cat error.log | claude
   ```
5. **让它自己去取** —— 告诉它"用 Bash 命令 / MCP 工具 / 读文件"自己获取上下文。比如："跑 `git log` 看看最近改了啥。"

新手建议：**先把 `@文件名` 和"贴截图"这两招练熟**，日常 80% 的场景够用了。

## 🔥 杀手锏：让 Claude 反过来"面试"你

这是**对非技术新手最友好的一招**，请重点掌握。

很多时候你心里有个模糊的想法，但说不清楚需求细节——这太正常了。这时候**别硬憋着写需求**，反过来让 Claude 采访你：

```text
我想做 [一句话描述]。用 AskUserQuestion 工具详细采访我。
问我技术实现、UI/UX、边界情况、顾虑和取舍。
不要问显而易见的问题，挖我没想到的难点。
问完之后，把完整的需求规格写进 SPEC.md。
```

它会像个产品经理一样，**一个个问题问你**，把你脑子里没想清楚的地方全挖出来。问完，它帮你生成一份 `SPEC.md` 需求文档。

**然后关键一步：写完 SPEC.md 后，开一个全新的会话去执行。**

为什么要开新会话？因为采访过程产生了一大堆来回对话，把工作记忆塞满了。新会话上下文干净，而你手里有一份白纸黑字、可以反复引用的规格——这比一段乱糟糟的聊天记录可靠太多了。

> 💡 什么样的 SPEC 最有用？**自包含的**：写明涉及哪些文件和接口、什么**不在**范围内、最后用什么**端到端的验证**来证明功能确实可用。

## 等等——什么时候反而该"模糊提问"？

不是所有时候都要精确。**探索阶段，模糊提问反而更好。**

当你想看看它**怎么理解一个问题**、想挖出你自己想不到的点子时，故意问得开放一点：

```text
这个文件，你觉得有什么可以改进的地方？
```

这种问法能让它**自由发挥**，经常能挖出你压根没注意到的问题。

**口诀：** 想要确定的结果 → 精确提问；想要意外的灵感 → 模糊提问。

## ✍️ 学员练习

挑一个你最近想让 AI 帮你做的事，用本篇的方法做两遍：

1. 先按你的老习惯随手写一句话指令。
2. 再用"四个模板"改写一遍（限定范围 / 指向信息源 / 引用现有模式 / 描述症状，挑适用的）。
3. 对比两次的产出差距。

然后试一次**杀手锏**：让它用 AskUserQuestion 采访你，生成一份 SPEC.md，再开新会话执行。体会那种"需求一下子清晰了"的感觉。
